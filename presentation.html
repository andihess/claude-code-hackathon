01 — The Mandate

The problem

Inbound IT support requests arrive through tickets, chat, and "urgent" emails to the CIO, and get hand-triaged by a human. Time-to-first-response is measured in hours nobody is proud of. We're building an agent that classifies, routes, and acts — not a chatbot.

02 — Architecture

The loop

Ingest → Classify → Enrich → Decide → Act → Log

  1. Ingest — raw request (subject, body, channel, requester).
  2. Classify — category + severity (P1–P4) + confidence.
  3. Enrich — knowledge-base lookup + system-of-record read.
  4. Decide — auto-resolve | route-to-queue | escalate.
  5. Act — the one write tool (route_ticket) creates/routes/resolves.
  6. Log — the full reasoning chain, replayable from the log alone.

03 — The Decision Contract

Every request produces exactly this JSON, validated before any action:

{
  "category": "access | network | hardware | software | security",
  "severity": "P1 | P2 | P3 | P4",
  "action": "auto_resolve | route | escalate",
  "target_queue": "string | null",
  "confidence": 0.0,
  "reasoning": "why this decision, in one short paragraph",
  "citations": ["kb-article-id or system-of-record ref, ..."]
}

Severity — explicit thresholds, not vibes

SevDefinition
P1Business-down or many users blocked; security incident in progress
P2One user fully blocked, or a team degraded, no workaround
P3Degraded but has a workaround
P4Low / cosmetic / request

"Urgent" in the body does not raise severity by itself — impact and scope do. Our fixture 004-urgent-language-cosmetic tests exactly this.

Action rules

  • auto_resolve — only access password resets, confidence ≥ 0.85, requester verified.
  • escalate — severity P1, OR confidence < 0.60, OR category == security. The safe default when unsure.
  • route — everything else, to the queue from CATEGORY_QUEUE_MAP.

04 — Tool Design

~5 tools, exposed via create_sdk_mcp_server(name="helpdesk", ...). Reads are pure; only route_ticket writes.

ToolTypePurpose
kb_lookupreadSearch KB/runbooks for a resolution
lookup_requesterreadRequester profile, VIP status, org unit
lookup_assetreadDevice/asset status tied to the request
check_queue_loadreadCurrent load per team queue, for routing
route_ticketwriteCreate/route/auto-resolve the ticket

Every tool description teaches what it does and does not do. Errors are structured, never a raw string:

{
  "content": [{"type": "text", "text": "Requester not found for email X"}],
  "is_error": true
}

05 — Validation-Retry Loop

Structured output via output_format bound to the schema above. On validation failure, the specific error is fed back to Claude and it retries — up to 3 times. Retry count and error type are logged per request.

Real retry example (bad output → specific error → corrected output) + a replayable reasoning-chain log excerpt — drops in once Track B's coordinator.py and logging/ land.

06 — Testing & Evals

12 labeled sample tickets across all 5 categories and all 4 severities, including deliberate boundary cases:

  • A clean, verified password reset (should auto-resolve)
  • The same request type, unverifiable — should not auto-resolve
  • A business-down P1 and an active-security-incident P1
  • A request that sounds urgent but is cosmetic (should stay P4)
  • Degraded-with-a-workaround cases (P3) vs. degraded-with-no-workaround (P2)

eval/run_eval.py scores: overall accuracy, per-category precision, and escalation rate (correct vs. needless). It fails gracefully — not with a crash — while the coordinator is still a stub, and a mocked-triage() smoke test validates the scoring math independent of Tracks A and B.
