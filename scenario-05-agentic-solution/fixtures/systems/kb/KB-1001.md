# KB-1001 — Reset a forgotten password

**Category:** access

## When this applies
User cannot sign in because they forgot their password or their account is locked
after too many attempts. This is a routine request, **not** a security incident,
unless there are signs of compromise (see KB-5001).

## Resolution
1. Verify the requester's identity via the requester directory (`lookup_requester`).
   Confirm `account_status == "active"` and `identity_verified == true`.
2. If verified, trigger a password reset and send a one-time link to the on-file email.
3. Advise the user to re-enroll MFA if prompted (see KB-1002).

## Auto-resolution eligibility
Eligible for **auto_resolve** when the requester is verified and confidence is high
(>= 0.85). Suspended or unverified accounts must be routed to `identity-access`, not
auto-resolved.

## Severity guidance
**P4** — a password reset is a routine request with an immediate, standard
remediation. Do **not** raise severity just because the user is currently locked out
or says it is urgent: the fix is fast, so the "blocked" state is transient, and the
reset itself is the workaround. Severity is set by impact and scope, not by tone or a
single blocked user.

Note the difference from a genuine P2 login failure: "can't log in at all" is P2 only
when there is **no** quick standard fix (e.g. account/identity broken, SSO down). A
forgotten or expired password does have a quick fix, so it stays **P4**. Treat higher
severity only when the lockout is a symptom of something broader — many users affected
(possible outage → P1/P2) or signs of compromise (security incident, see KB-5001).
